Authentication Analysis — Brute Force & Password Spraying
Windows EventID 4625/4740 · Active Directory · Linux auth logs
EventID 4625EventID 4740Password SprayT1110T1110.003

⚔ What is Brute Force?

A brute force attack fires a high volume of password guesses against a single account. The attacker systematically tries every combination. Windows logs each failure as EventID 4625 and triggers a lockout after the configured threshold (EventID 4740).

Typical indicator: >50 failed attempts per minute from the same source IP.

🎯 What is Password Spraying?

Instead of targeting one account, the attacker tries one common password (e.g., Summer2024!) across many accounts. This evades lockout policies by staying below the threshold per account.

Key signal: same source IP, low fail count per account, many different target usernames in the same 30-minute window.

💀 Why is it Dangerous?

Successful credential access provides a network foothold. In Active Directory environments, even a low-privilege account can be leveraged for lateral movement, Kerberoasting, or Pass-the-Hash. It is the #1 initial access vector in enterprise breaches (DBIR 2024).

⚠
Key Difference: Brute Force = 1 target + many attempts · Password Spray = many targets + few attempts each

Query 1 — Brute Force Detection (EventID 4625)

SPL · Brute Force · Windows Security Logs index=* · earliest=-1h
index=* sourcetype=WinEventLog:Security EventCode=4625 earliest=-1h | stats count as failed_attempts, dc(src_ip) as unique_ips, values(src_ip) as source_ips, first(_time) as first_seen, last(_time) as last_seen by Account_Name, Logon_Type | where failed_attempts > 10 | eval duration_min = round((last_seen-first_seen)/60,2) | eval attack_type = if(unique_ips==1,"Brute Force","Distributed") | sort -failed_attempts | table Account_Name, failed_attempts, attack_type, source_ips, duration_min, Logon_Type
ComponentExplanation
EventCode=4625Windows failed authentication — fires on every incorrect password attempt
dc(src_ip)Distinct count of source IPs — 1 IP = Brute Force · multiple = Distributed Attack
failed_attempts > 10Threshold filter — tune to >50 for lower noise in large environments
eval attack_typeAuto-classify: single IP = Brute Force · multiple = Distributed Attack

Query 2 — Password Spraying Detection

SPL · Password Spray · Windows + Linux earliest=-30m
index=* (sourcetype=WinEventLog:Security EventCode=4625) OR (sourcetype=linux_secure "Failed password") earliest=-30m | stats count as total_attempts, dc(Account_Name) as targeted_accounts, values(Account_Name) as account_list by src_ip | where targeted_accounts > 5 AND total_attempts < (targeted_accounts * 3) | eval spray_score = round(total_attempts/targeted_accounts,2) | eval verdict = if(spray_score<3, "HIGH - Password Spray","POSSIBLE Spray") | sort -targeted_accounts
ComponentExplanation
dc(Account_Name)Count of distinct accounts targeted from one IP — core spray signal
spray_scoreAttempts-per-account ratio — closer to 1.0 = higher confidence password spray
total < (accounts * 3)Spray logic: few attempts per account keeps each below lockout threshold

Red Flags: 02:17 AM (after-hours) · 87 attempts in 4 minutes · Account locked · Logon_Type=3 (network)
🔍 Phase 1 — Alert Validation ▾
Verify failed_attempts count — above threshold? (default: 10 in 5 min)
Check if EventID 4740 (Account Lockout) is also present
Check time — is it outside business hours?
📊 Phase 2 — Evidence Collection ▾
Geolocate source IP — does it match the user's expected country/city?
Check if same IP appears in Firewall, Proxy, or VPN logs
Search EventID 4624 — any successful login AFTER the failures?
Verify account in Active Directory — exists, enabled, last login date
🌳 TP / FP Decision Tree
Are there 10+ failed attempts in under 5 minutes?
✗ No → Likely FP (user forgot password) — document and close
✓ Yes → Continue
Is there a successful login (EventID 4624) following the failures?
✓ Yes → HIGH CONFIDENCE True Positive — Escalate immediately
✗ No → Continue
Is source IP from an unexpected country or location?
✓ Yes → True Positive — Block IP + Escalate
✗ No → Likely legitimate user — contact account owner
🚨 Phase 3 — Response Actions (TP Confirmed) ▾
Disable the targeted account and notify the account owner
Block source IP in Firewall / WAF
If successful login found: investigate post-auth activity (processes, file access)
Open ticket in Splunk ES and write Response Notes (NOT Audit Notes)
